# Requirements Clarification Checklist

> 需求澄清检查清单，确保需求在文档化前被充分理解

---

## 1. Problem Space（问题空间）

### 1.1 问题定义
- [ ] 我们要解决什么问题？
- [ ] 谁受到这个问题的影响？
- [ ] 不解决这个问题会有什么后果？
- [ ] 这个问题有多紧急？

### 1.2 用户理解
- [ ] 目标用户是谁？
- [ ] 用户的核心痛点是什么？
- [ ] 用户当前如何解决这个问题？
- [ ] 用户对解决方案有什么期望？

---

## 2. Solution Space（方案空间）

### 2.1 期望行为
- [ ] 功能的核心行为是什么？
- [ ] 触发条件是什么？
- [ ] 正常流程是什么样的？
- [ ] 异常情况如何处理？

### 2.2 范围边界
- [ ] 明确包含什么功能？
- [ ] 明确不包含什么功能？
- [ ] 有哪些假设前提？
- [ ] 第一版 vs 后续版本的边界？

### 2.3 约束条件
- [ ] 技术约束（平台、性能、安全）？
- [ ] 业务约束（法规、政策）？
- [ ] 时间约束（截止日期）？
- [ ] 资源约束（预算、人力）？

---

## 3. Verification Space（验证空间）

### 3.1 验收标准
- [ ] 如何判断功能正常工作？
- [ ] 核心场景的验收条件是什么？
- [ ] 边界情况有哪些？
- [ ] 错误场景如何验证？

### 3.2 最小可行版本
- [ ] 最小可用版本包含什么？
- [ ] 哪些功能可以后续迭代？
- [ ] 什么是不可妥协的核心？

---

## 4. Context Space（上下文空间）

### 4.1 依赖关系
- [ ] 依赖什么现有功能？
- [ ] 什么功能依赖这个？
- [ ] 需要什么外部系统/服务？
- [ ] 数据依赖是什么？

### 4.2 优先级
- [ ] 相对于其他需求的优先级？
- [ ] 是否阻塞其他工作？
- [ ] 有没有时间敏感性？

### 4.3 相关文档
- [ ] 是否已有相关 PRD/Spec？
- [ ] 是否有设计稿/原型？
- [ ] 是否有历史讨论记录？

## 6. Common Clarification Questions（常用澄清问题）

### 范围相关
```
- 你提到了 {X}，这是否包含 {Y}？
- {Z} 是这个需求的一部分，还是应该单独处理？
- 有什么是明确不需要做的？
```

### 行为相关
```
- 当用户 {操作} 时，期望发生什么？
- 如果 {异常情况} 发生，应该如何处理？
- 这个操作需要什么权限？
```

### 验收相关
```
- 你如何知道这个功能是正常工作的？
- 什么情况下你会认为这个功能失败了？
- 最小可接受的版本是什么样的？
```

### 优先级相关
```
- 这个功能的紧急程度如何？
- 如果只能完成一部分，哪部分最重要？
- 这个是否阻塞其他工作？
```
